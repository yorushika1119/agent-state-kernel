## KMS 与 Kernel 的分工

按设计文档的定位，应该这样理解：

- `KMS`：上层调度和解释层  
  负责“收到一个事件或请求以后，应该怎么理解、能不能写、属于哪类状态更新”
- `Kernel`：底层状态内核  
  负责“session / run / event log / reducer / views 这些状态怎么存、怎么更新、怎么查询”

通俗一点说：

- `KMS` 更像“调度大脑”
- `Kernel` 更像“状态账本”

所以这次打断机制里：

- **Hermes / gateway** 负责接收用户新消息，并把“有新请求了”传上来
- **KMS + Kernel** 负责决定：
  - 这是继续当前 session
  - 还是新建任务
  - 如果继续当前 session，旧 run 现在是否失效
- **Thinker** 只负责按当前 run 干活，不负责自己决定调度策略

## 打断机制是怎么工作的

当前默认行为：

1. 用户发第 1 条请求
2. gateway 调 `dispatch-user-message`
3. kernel 创建或命中一个 session，生成 `run_1`
4. thinker 开始执行 `run_1`
5. 用户在 `run_1` 还没结束时，又发第 2 条请求
6. gateway 再次调 `dispatch-user-message`
7. kernel 判断“这是同一 session 的新请求”，返回 `interrupt_and_replan`
8. kernel 把这个 session 的 `active_run_id` 从 `run_1` 切换到 `run_2`
9. Hermes 通知旧 agent 中断，转去执行 `run_2`
10. 如果旧 `run_1` 之后还想继续往 kernel 写事件，会被拒绝，错误原因是：
    `Stale thinker run: submitted run_1, active is run_2`
11. 新 `run_2` 执行完成后，gateway 再调 `/kms/complete-run`
12. kernel 把 `active_run_id` 清空

另外，kernel 现在会在旧 run 上追加一个正式的 `RunInterrupted` 事件。
它的作用不是恢复执行，而是把“旧 run 被哪一个新 run 顶掉了”正式记入事件日志和视图。

这里最关键的一点：

这个“打断”不是操作系统层面的强杀线程，而是**调度权切换**。

也就是：

- kernel 收回旧 run 的写入资格
- Hermes 停止继续推进旧 run
- 就算旧 run 晚一点把结果交上来，也写不进当前 session

所以它看起来像 Codex 的打断体验，本质上是“状态和调度层打断”，不是“底层执行器可抢占”。

## `step` 和 `run` 的区别

这个点之前容易混。

可以这样理解：

- `run`
  是“一次完整请求的执行轮次”
  比如用户发一次消息，kernel 分配一个 `run_id`
- `step`
  是这次请求内部计划里的某一个步骤
  比如“先搜索资料，再整理结论”，这是两个 step

所以：

- 一个 `run` 里，可能有多个 `step`
- 一个 `step` 不是一个新的 `run`
- 当前打断机制切换的是 `run`

现在的边界是：

- run 级打断已经稳定
- step 级推进在 kernel 里已经成立
- Hermes 侧现在已经能在有进行中 todo 时直接上报该 todo 的 `step_id`
- 只有没有进行中 todo 的场景，kernel 才还会做一层 `current_step` 映射

## 下一步建议

建议按这个顺序继续：

1. 如果你以后想支持“不中断旧任务，只排队新请求”，再设计 `queue_after_current`
2. 如果你以后想支持“新问题问完以后，再回来继续旧任务”，再设计旧 run 的恢复机制
3. 如果你以后想把 step 语义做得更强，再把 Hermes 的 plan / todo 生命周期和 kernel 的 step 生命周期进一步对齐

当前不建议马上做：

- 同一 session 内双 thinker 并行
- 旧 run 自动恢复执行
- 过度复杂的容错和兜底

原因很简单：

你现在最核心的“像 Codex 一样打断当前任务，转去处理新任务”已经成立了。
下一步应该先把这套最小闭环打磨稳定，而不是过早做复杂扩展。
